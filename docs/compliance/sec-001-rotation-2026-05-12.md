# SEC-001 OpenAI key rotation — 2026-05-12

## What

The OpenAI API key flagged in
[`docs/compliance/HARDENING_AUDIT_2026-05-11.md`](./HARDENING_AUDIT_2026-05-11.md#sec-001-live-api-key-in-env-on-disk)
(`sk-proj-arT5...`) has been rotated.

## Steps taken

1. New key generated at https://platform.openai.com/api-keys (named
   `paseo-daemon`).
2. Stored in claude-vault via `python C:/Dev/tools/vault/vault.py set
openai api_key <new-key>`. Verified present in vault listing.
3. `packages/server/.env` `OPENAI_API_KEY=` cleared (had previously been
   blanked out, confirmed empty on rotation day).
4. Old key revoked at https://platform.openai.com/api-keys.
5. `.env.example` + `.env.production.example` updated to point operators
   at vault / Docker secret instead of pasting raw keys into env files.

## How code reads the key now

| Environment      | Source                                                                                                                |
| ---------------- | --------------------------------------------------------------------------------------------------------------------- |
| Local dev        | `vault get openai api_key` → exported as `OPENAI_API_KEY` in shell or read directly by `secret-loader.ts`             |
| Docker (compose) | `/run/secrets/OPENAI_API_KEY` (declared in `docker-compose.prod.yml` + `docker-compose.deploy.yml` `secrets:` blocks) |
| CI               | Repository secret `OPENAI_API_KEY` (set via GitHub Actions UI)                                                        |

`packages/server/src/server/secret-loader.ts` `loadSecret("OPENAI_API_KEY")`
returns the first match in this priority order: Docker secret file → env
var. Callers don't need to know the source.
